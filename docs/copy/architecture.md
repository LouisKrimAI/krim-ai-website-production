## 4 · The integration fabric
EYEBROW: HOW IT CONNECTS
H2: 40+ connectors. Event-driven, with a fallback for everything older.
COPY: Core banking, servicing, CRM, telephony, messaging, payments, bureaus — KrimOS connects through 40+ pre-built adaptors. Where systems support it, integration is event-driven and real-time; where they don't, it falls back to batch. Either way, your systems stay canonical.
PRESENTATION: a connector map — KrimOS reaching into named system categories; live vs batch shown subtly.

## 5 · One architecture, every market and domain
EYEBROW: PORTABILITY
H2: The architecture never changes. The rulebook does.
COPY: The same runtime runs in every market and every domain. Only Krim-Fabric — the layer that holds the rules — changes. Move to a new jurisdiction or a new industry and you inherit the same runtime, the same audit trail, the same governance, with the local rules already in place.
PRESENTATION: a single architecture with a swappable "rules" layer; switching it re-tints the system without changing its shape.
